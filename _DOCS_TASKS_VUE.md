# Vue binding documentation tasks

Audit scope: `packages/vue/README.md`, `packages/vue/package.json`, every page in `docs/vue/`, all files in `packages/vue/lib/`, and `packages/vue/tests/index.test.ts`. The Vue suite passes (4 tests).

## Актуализация документации

- [x] Fix the fork-scope plugin setup in both `packages/vue/README.md` and `docs/vue/index.md`. They show `app.use(EffectorScopePlugin, { scope })`, but `effector-vue@23.1.1` exposes `EffectorScopePlugin(config): Plugin`; the documented call treats the factory itself as a Vue function plugin and leaves `app.config.globalProperties.scopeName` unset. Use `app.use(EffectorScopePlugin({ scope }))`. The current suite avoids the public plugin by defining a different local `EffectorScopePlugin(scope)` helper, so add a test for the exact documented setup.

- [x] Correct the route-params example in `packages/vue/README.md`. `useUnit(userRoute.$params)` returns a Vue `Ref`, so `const { id } = useUnit(...)` attempts to take `id` from the ref object rather than from its value. Keep the ref (`const params = useUnit(...)`) and rely on template unwrapping, or derive `id` from `params.value`. Add a typechecked documentation example; none of the four Vue tests reads route stores directly through `useUnit`.

- [ ] Document the complete `createRouteView` / `createLazyRouteView` contract. Both `CreateBaseRouteViewProps` and `RouteView` support `route: Route | Router` and nested `children`, while the two API pages describe only an ordinary route and omit `children`; the `Outlet` page relies on that undocumented field. The only related test merely verifies that a lazy view accepts a nested router without throwing. Add eager and lazy nested-router rendering tests plus `children`/`Outlet` tests that match the guide.

- [ ] Narrow or enforce the claimed `RouterProvider` requirement. `docs/vue/router-provider.md` says all views, links, and composables must be inside the provider, and `docs/vue/create-routes-view.md` says `createRoutesView` works only there. In the implementation, router injection is required only by `useRouterContext` and its consumers (`useRouter`, `useLink`, and `Link`); `createRoutesView`, `useOpenedViews`, and `useIsOpened` subscribe directly to the supplied Effector units, and `Outlet` uses a separate outlet injection. Decide whether provider presence is an intended invariant: either document the actual boundary or validate it in every advertised API. Existing tests always mount through `RouterProvider`, so add provider-free/error-path coverage for the chosen contract.

- [x] Complete the `useLink` and `useRouter` return contracts. `useLink.build` accepts both `(params, query)` and appends a serialized query, and `useLink` throws when the route is absent from `router.knownRoutes`; `docs/vue/use-link.md` documents neither the second argument nor the failure condition. `useRouter()` exposes the router unit shape `{ query, path, activeRoutes, onBack, onForward, onNavigate }`, but `docs/vue/use-router.md` shows only `path` and `query`. Document the callable/state fields and their Vue ref unwrapping rules. There are no direct tests for `useLink`, `useRouter`, or `useRouterContext`.

- [x] Reconcile route-priority wording across `createRoutesView` and `useOpenedViews`. `docs/vue/create-routes-view.md` says that whenever several routes are active the last listed view wins, but `useOpenedViews` first removes an active parent in favor of its active child; declaration-order priority applies only after that filtering. State the two-stage rule explicitly. The current route-switching test never has multiple routes active, and there is no nested-parent priority test.

## Модификация поведения

- [ ] Make the `Link` page match the runtime and type surface. `docs/vue/link.md` lists only `to` and `params`, but the component also implements `query`, `replace`, `target`, `onClick`, arbitrary anchor attrs, cancellation through `preventDefault`, modified-click passthrough, and non-`_self` target passthrough. Conversely, the page says params are required for parameterized routes, while the exported `Link` component declares `to: Route<any>` and optional `params: any`; the conditional generic `LinkProps` type is exported but is not applied to the component declaration, so Vue consumers do not receive that promised enforcement. Document all supported behavior and either wire `LinkProps` into `Link` or state the real typing contract. The suite covers only an unmodified click with params/query; add type tests and runtime tests for replace, attrs, cancellation, modifiers, and targets.

- [ ] #57 RFC: preserve shared layout instances across `withLayout` route transitions. Decide whether layout identity/preservation belongs to the shared RouteView contract or to a Vue-specific primitive; moving between two views in one layout group must not remount the layout.

## Контроль качества

- [ ] Add documentation-conformance coverage for the remaining untested public behavior: lazy component resolution/loading fallback/layout, direct `createRouteView` layout, nested and multi-level `Outlet`, `useIsOpened` for both a route and a router, `useOpenedViews` parent filtering, `RouterInjectionKey`/missing-provider behavior, and preservation of `children` by `withLayout`. The current four tests cover route switching/fallback, one normal `Link` click, one `withLayout` render, and only construction (not rendering) of a lazy nested-router view.
